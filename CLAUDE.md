# Coleitai Blog — 프로젝트 헌법

## 1. 프로젝트 의도

이 저장소는 **누구나 (중학생도) 본인만의 SEO 블로그를 만들 수 있는 "밀키트" 템플릿**이다.
- Next.js 15 + Turso(LibSQL) + Vercel 기반
- Lighthouse 400점 (성능·접근성·모범사례·SEO 각 100점)
- Claude Cowork에서 클론 후 대화만으로 커스터마이징 가능

## 2. 커뮤니케이션 규칙

- **항상 한국어로 보고한다.** 영어 전용 용어(SEO, Lighthouse, Prisma 등)만 예외.
- **중학생도 이해할 수 있는 수준으로 설명한다.** 전문 용어를 사용할 때는 반드시 괄호 안에 쉬운 설명을 붙인다.
- 작업 완료 후: 어떤 파일을 수정/생성했고, 무엇이 바뀌었는지 3줄 이내로 요약한다.

| `src/config/brand.config.ts` | 로고, 저작권 정보 |
| `src/config/features.config.ts` | 기능 on/off 토글 |

| `src/app/[locale]/page.tsx` | 홈페이지 디자인 수정 |

## 4. 절대 수정 금지 파일 (위험 영역)

| 경로 | 이유 |
|------|------|
| `prisma/schema.prisma` | DB 구조 변경 → 데이터 손실 위험 |

## 5. 검증 기준 (모든 변경 후 확인)

## 6. 기획 워크플로우 (WWH 프레임워크)

사용자가 "OO 만들어줘", "OO 기획해줘" 같이 새 기능을 요청하면 아래 순서를 따른다.

### 6-1. 기획 모드: Why → What → How

| 단계 | 비중 | 질문 | 핵심 |
|------|------|------|------|
| **Why** | 30% | 왜 해야 하는가? | 문제 정의, 타겟 유저, 현재→목표 수치 |
| **What** | 40% | 뭘 만드는가? | 솔루션 한 줄 요약, MVP 범위, **Not-doing list (필수)**, KPI |
| **How** | 30% | 어떻게 만드는가? | 기술 스택, 구현 방식, 리스크 |

> **규칙:** What 없이 How를 시작하지 않는다. Not-doing list가 없으면 기획을 반려한다.

### 6-2. 설득 모드: Why → How → What (Golden Circle)

README, 랜딩 페이지, 소개 글 등 **사람을 설득하는 문서**를 작성할 때:
- **Why** — 왜 존재하는가 (비전/신념)
- **How** — 어떻게 다른가 (차별화/방법론)
- **What** — 뭘 제공하는가 (제품/결과)

### 6-3. 검토 체크리스트

승인 요청 전 반드시 확인:
- [ ] 문제가 구체적인가? (추상적이면 반려)
- [ ] 타겟 유저가 명확한가?
- [ ] MVP 범위가 적절한가? (너무 크면 쪼갠다)
- [ ] Not-doing list가 있는가?
- [ ] 기존 코드/기능과 충돌하지 않는가?

### 6-4. 승인 → 실행

사용자 승인 후 개발 규칙:
- Next.js + React + TypeScript + TailwindCSS 스택 유지
- 승인된 범위를 임의로 변경하지 않는다
- 완료 후 한국어로 보고한다

## 7. 하네스 엔지니어링 (작업 환경 설계)

> 핵심 철학: 좋은 프롬프트보다 **좋은 작업 환경**이 더 중요하다.

### 7-1. 작업 원칙

1. **단일 진입점**: 실행 스크립트는 하나. 환경변수/설정/입력으로 제어한다.
2. **검증 루프**: 테스트, 로그, 리포트가 진실의 원천이다. 감이 아니라 결과로 판단한다.
3. **문서 기반 상태 관리**: worklog.md에 작업 내용을 누적 기록한다.
4. **컨텍스트 = 작업 메모리 예산**: 불필요한 정보를 컨텍스트에 넣지 않는다.

### 7-2. 반복 루프

```
1. issue.md에 문제 정의
2. hypothesis.md에 가설 3개 이상 작성
3. scripts/run.sh로 일관되게 실행
4. 결과 비교
5. worklog.md에 기록 (변경사항, 명령어, 증거, 결론)
6. 재현 실패 시 가설 갱신
```

### 7-3. worklog.md 기록 규칙

모든 주요 작업 완료 후 `worklog.md`에 다음 형식으로 기록한다:

```markdown
## YYYY-MM-DD — 작업 제목
- **변경**: 수정한 파일과 내용
- **이유**: 왜 이 변경이 필요했는지
- **검증**: 어떻게 확인했는지 (빌드 성공, 테스트 통과 등)
```

## 8. 에러 대응 규칙

에러가 발생하면 다음 순서를 따른다:

1. **한국어 번역**: 에러 메시지를 한국어로 번역해서 사용자에게 보여준다
2. **원인 분석**: 왜 발생했는지 1-2줄로 설명한다
3. **해결**: 코드를 수정하고 빌드/타입체크를 다시 실행한다
4. **검증**: 해결되었음을 확인하고 보고한다

> 에러 메시지를 그대로 보여주지 않는다. 항상 한국어로 설명을 붙인다.

## 9. 기술 스택 & 성능 규칙

### 기술 스택

| 기술 | 용도 |
|------|------|
| Next.js 15 (App Router) | 프레임워크, SSG 우선 |
| React 19 | UI |
| TypeScript | 타입 안전성 |
| TailwindCSS 4 | 스타일링 |
| Prisma + LibSQL | DB ORM (Turso) |
| Gemini API | AI 콘텐츠 생성 |
| Vercel | 호스팅/배포 |

### 성능 규칙

- **SSG 우선**: 콘텐츠 페이지는 정적 생성
- **이미지**: `next/image` 사용, alt 텍스트 필수
- **폰트**: `next/font` 사용
- **Core Web Vitals**: LCP, INP, CLS 우수 등급 유지
- **번들 크기**: 불필요한 라이브러리 추가 금지

### SEO 규칙

- 모든 블로그 글에 `Article` / `BlogPosting` schema.org JSON-LD 포함
- `rel="canonical"` 정확히 설정
- `robots.txt`, `sitemap.xml` 자동 생성
- 메타 태그 (title, description) 모든 페이지에 설정

## 10. Superpowers 개발 워크플로우

> 출처: obra/superpowers — 구조화된 AI 개발 방법론

코드를 바로 작성하지 않는다. 다음 단계를 거친다:

1. **브레인스토밍**: 문제를 이해하고 접근 방식을 탐색한다
2. **설계**: 구현 계획을 세우고 사용자 승인을 받는다
3. **작업 분해**: 큰 작업을 작은 단위로 쪼갠다
4. **구현**: 한 번에 하나의 작업을 완료한다
5. **리뷰**: 완료된 코드를 검토한다

### 핵심 원칙
- 테스트 주도 개발 (TDD): Red → Green → Refactor
- 한 번에 하나의 작업에 집중한다
- 변경 전에 기존 코드를 먼저 읽는다
- 사용자에게 보이지 않는 곳에서 복잡성을 숨기지 않는다

## 11. Context7 — 최신 문서 참조

## 12. 전문 지식 시스템 (knowledge/)

사용자가 `/admin/knowledge`에서 업로드한 마크다운 파일은 `knowledge/` 폴더에 저장된다.
AI 콘텐츠를 생성할 때 이 파일들을 참고하여 사용자의 전문성을 반영한다.

### 지식 파일 활용 규칙
- `knowledge/` 폴더의 .md 파일을 콘텐츠 생성 시 컨텍스트로 포함한다
- 사용자의 전문 영역, 경험, 데이터를 존중하고 활용한다
- 지식 파일의 내용을 그대로 복사하지 않고, 자연스럽게 녹여서 글을 작성한다
